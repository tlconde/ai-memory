Agent Memory Protocol — substrate-level specification

The durable layer between you and every AI you use.

Models change. Agents change. Tools change. Your knowledge, your preferences, your work-in-progress, and your skills shouldn't. AMP is the substrate that survives all of them — making information captured in any AI surface reachable from every other, with corrections that propagate cleanly and memory that mutates as a first-class operation.

§ 01 · The problem

Information stranded, corrections lost, work repeated.

Every AI you use builds its own private picture of you. None of them talk to each other. You correct one; the others stay wrong. You teach Claude something useful; Cursor never learns it. You build a skill in Hermes; Codex never sees it.

AMP exists because the fragmentation is structural, not accidental. Each surface optimizes for its own context window, its own session, its own vendor account. The substrate beneath them — the durable layer that ought to remember — doesn't exist yet.

01

State is stranded

Cursor saves project preferences. Claude Code saves them separately. Codex separately again. The same fact is captured in each surface, often in slightly different forms.

02

Surface memory is volatile

Compaction loses information that mattered. Users start fresh sessions and re-explain everything they already taught the system.

03

Corrections don't propagate

Correcting one surface doesn't update the others. Users correct each surface, every time, forever.

04

Procedures don't follow

A skill authored in Hermes is unavailable in Claude Code. Cross-harness skill portability requires manual file copying every time.

05

Defaults never improve

When a system mis-classifies a fact, the correction trains nothing. The same mistake recurs in every future session.

∴

The substrate is missing

A durable layer between models, harnesses, and storage. Replaceable above and below; persistent across both. AMP fills this gap.

§ 02 · Architectural commitments

Seven principles that shape everything.

These commitments came from iterative pressure-testing. They survived multiple rounds of user correction. They are restated at the top because principles stated once drop between conversations; principles in the lead framing survive.

i
The three-layer model: Model · Harness · Substrate
Models are replaceable. Harnesses are replaceable. Substrate is the durable layer that survives both. Storage backends plug into the substrate; multiple backends may coexist; the substrate orchestrates them.
ii
Skills as the empowerment surface
Skills should grow without bound. AMP stores a canonical procedural artifact, then compiles it into harness-native emitted files. Cursor's .mdc and Claude's SKILL.md are outputs, not competing sources of truth.
iii
Knowledge, procedures, and models are structurally different
Knowledge has mutation semantics. Procedures have a scaffold-fill-check-version lifecycle. Models interpret both. AMP orchestrates their interconnection — it does not collapse them into one shape.
iv
Determinism where possible
Schema-based operations beat freeform. Typed slots beat untyped properties. Explicit confidence beats implicit hand-waving. Where behavior can be made deterministic, it should be.
v
Single-user knowledge scope (v1)
No cross-user knowledge sharing in v1. The shareable layer is the protocol itself and skills (with personalizations removed). Knowledge stays single-tenant.
vi
AMP defines — backends declare — users informed
AMP defines the full standard. Each backend's adapter declares capability coverage. Missing capabilities surface to users as informed gaps — not as failures, not as silent omissions.
vii
Falsifiability invariant
Every behavior claim has a concrete falsifiable test. Vague claims are insufficient. Specific testable claims are the standard, throughout.
§ 03 · High-level architecture

Four layers, with AMP at the centre.

The system has four layers. AMP is Layer 2 — the substrate. Layer 0 (models) and Layer 1 (harnesses) sit above. Layer 3 (storage backends) sits below. AMP orchestrates the interconnection.

High-level architecture — four layers

Replaceability is the load-bearing claim. Swap Claude for Gemini — the substrate doesn't care. Swap Hermes for Cursor — the substrate doesn't care. Swap gbrain for Mem0 — the substrate still serves the same operations, with capability gaps surfaced to the user.

§ 05 · The frame layer

Three kinds, each with distinct semantics.

The substrate's knowledge layer is built on three frame kinds. Every piece of stored knowledge is one of these. The kinds are operationally distinct — they have different mutation rules, different retrieval patterns, different validity models.

Episodic
mutation: append-only
validity: never expires
retrieval: time-windowed

Events that happened. Immutable once written. Conversations, observations, signals, corrections, transitions.

"Met Aurora on May 22 at Prosus event"
Semantic
mutation: supersedes
validity: valid_from / valid_until
retrieval: scope-filtered

Stable facts about the world. Mutable through supersedes. Preferences, identity facts, time-bounded claims about people and projects.

"Aurora works at company X (since June 2024)"
Crystal
mutation: refine conditions
validity: indefinite, refinable
retrieval: confidence-weighted

Falsifiable claims about regularities of the environment or of designed systems. Not for human behavior. Refined through conditions, not replaced.

"Throwing glass on hard surfaces breaks it — refined: tempered glass at low velocity is exception"

Profile is NOT a fourth kind. Profile slots like reading_list, active_intent, and relationships are saved queries over these three kinds plus the runtime registry and graph. They look like typed storage to the user; they are computed views underneath. This kept the kind count at three after rigorous pressure-testing.

§ 06 · Low-level architecture

Inside the substrate: storage, inference, consolidation, propagation.

The substrate is itself decomposed into four sub-layers. Each has a specific responsibility. Together they implement the durable layer.

SL · 1
Storage
runtime + knowledge

The runtime store holds operational state: active intent, harness health, propagation queue, sync state, unprocessed signals. Cache-like.

The knowledge store holds processed frames, the graph, embeddings, profile schema, procedural registry. Durable.

SL · 2
Inference
replaceable classifiers
  • Kind classifier
  • Curation_mode classifier
  • Entity extractor
  • Slot router
  • Edge type inferrer

v0: rule-based. v1: corrections train per-user adjustments. v2: small fine-tuned models.

SL · 3
Consolidation
the overnight processor

Runs on idle, schedule, or demand. The biological-sleep analogue.

  1. Drain unprocessed episodic
  2. Extract entities
  3. Wire typed edges
  4. Refresh embeddings
  5. Detect contradictions
  6. Decay / prune
  7. Write to knowledge
  8. Clear runtime
  9. Session manifest
SL · 4
Propagation
cross-harness skill flow

Watches the procedural artifact registry. When the user authors or updates a skill, propagation compiles the canonical AMP artifact into every verified compatible harness's from-amp/ directory.

Cursor target uses .mdc extension and flat naming. Other harnesses use folder-per-skill.

§ 07 · The unified knowledge graph

One graph. Three curation modes.

The earlier draft had two databases — one personal, one for LLM-curated knowledge. The current architecture is cleaner: a single unified knowledge graph, where every node carries a curation_mode property.

personal
scope rule: strict (Invariant 1)
default for: user-symbiosis content

Facts about the user's world. The personal observation store. Scope never inferred upward.

llm_curated
scope rule: distillation allowed
default for: background knowledge

Karpathy-style LLM wiki. Reference material the system curates for itself. Reframable, distillable.

shared
scope rule: opt-in by user
default for: explicit promotion only

Accessible from both consumer contexts. Rare; explicit user opt-in. Promotion requires action.

The router is now a query filter, not a multi-backend dispatcher. Reads default to filtering by current consumer context. Writes go through curation-mode-specific gates. Cross-mode references work naturally because edges span freely in a unified graph.

§ 08 · Procedures — adopt, don't invent

AMP doesn't define a skills format. The standard exists already.

AMP uses a compiler model for procedures. The canonical AMP artifact is the durable source; Cursor .mdc, Claude Code SKILL.md, and future harness-specific files are emitted artifacts. v1 verifies Cursor, Claude Code, and Hermes at the filesystem level; Codex, Gemini, Windsurf, and other harness placements remain out of v1 verified scope until tested.

What AMP adds

Beyond the standard name and description fields, AMP adds:

  • amp_artifact_version — version pinning
  • amp_compatibility — required frame kinds, profile slots, curation modes
  • harness_compatibility — declares which harnesses can execute this skill
  • harness_overlays — per-harness behavior tweaks (Cursor globs, Claude Code hooks, etc.)
  • Relationship metadata: extends, required_by, conflicts_with

The from-amp invariant

AMP writes only to from-amp/ subdirectories within each harness's skill directory. User-authored content lives outside; AMP never touches it. This is Invariant 4 — protocol-level, not implementation detail.

§ 09 · The RL feedback loop

Defaults that actually improve.

Corrections are training signal. When AMP infers a default (which kind, which curation_mode, which entity, which slot) and the user overrides it, the disagreement is recorded with full provenance. The corpus of corrections becomes the dataset for retraining the inference layer's classifiers.

v0 ships rule-based defaults. v1 adds the feedback loop: per-user adjustments learned from correction frequency. v2 could ship small fine-tuned models per inference task.

The architecture absorbs this cleanly: the inference sub-layer is a first-class substrate component (§ 06). Each classifier is replaceable. Versioning is per-classifier. The training cron runs weekly or on-demand.

This is what makes AMP compound. Without the feedback loop, AMP is static infrastructure. With it, AMP gets sharper for each user over time — the same mistake doesn't recur. The longer you use it, the better its defaults match how you think.

§ 10 · Deployment shapes

Three ways to run AMP.

Shape A · Local-only

Substrate runs locally with consolidation daemon. v1 verified offline: Cursor, Claude Code, and Hermes filesystem adapters (path guards, emit, propagation E2E). Live harness session load and other local harnesses remain PROVISIONAL until placement and load behavior are tested. Cloud surfaces get briefing-only mode.

Network: none required

Shape B · Hybrid

Same as Shape A, plus AMP exposes a public OAuth-protected MCP endpoint. Adds claude.ai web and Cowork coverage. ChatGPT still gets briefing-only.

Network: public-reachable URL (VPS, Tailscale Funnel, Cloudflare Tunnel)

Shape C · Briefing-only

No daemon, no fs-watcher, no MCP server. AMP runs as a CLI library. User triggers briefing generation on demand. Reduced functionality — no consolidation, no inference training.

Use case: cloud-only users, no daemon preference
